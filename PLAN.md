### 律师行业 AI Agent 助手需求文档 (PRD)

#### 1. 核心目标
打造一个安全、专业、高效的法律 AI 助手。通过私有知识库与外部互联网信息的结合，为律师提供精准的法律条文检索、案例分析、文书草拟及网络调研服务。

#### 2. 用户角色与核心流程
* **用户角色：** 执业律师、法务人员、律师助理。
* **核心流程：**
    1.  **注册/登录：** 确保数据隔离与隐私安全。
    2.  **知识库构建：** 用户上传本地法律文件（PDF、Word 等），系统后台排队处理，前端实时展示解析与向量化状态（如：排队中、解析中、入库成功、失败）。
    3.  **Agent 交互：** 用户通过对话框发起请求。Agent 自动规划任务，调用检索、搜索等工具，最终输出带有溯源引用（Citations）的回答或长篇报告。

#### 3. 核心功能模块与工具 (Tools) 需求
* **私有知识库管理 (RAG 基础)：**
    * 支持多格式（PDF、DOCX、TXT）。
    * **核心要求：** 必须支持复杂版面分析（法律文档常有双栏、表格、长篇条款的层级结构）。
* **Agent 智能工具箱：**
    * **知识库检索工具 (Knowledge Retrieval)：** 在用户的私有库中进行语义及关键词混合检索。
    * **网络搜索工具 (Web Search)：** 检索最新公布的法律法规、实时新闻或公开的裁判文书（可接入 Google Search 等 API）。
    * **网页深度抓取工具 (Web Scraper)：** 针对搜索到的特定网页或公开案例库页面，提取正文内容进行深度分析。
    * **文书/报告撰写工具 (Report Writer)：** 根据收集到的所有信息，按法律文书标准格式生成长篇报告，支持导出为 Word 或 PDF。

### 技术架构方案设计

为了实现上述需求，系统需要采用典型的**前后端分离 + 异步任务队列 + Agent 编排 + 复杂 RAG 流水线**的架构。

#### 1. 基础设施与基础服务
* **前端 (Web)：** React / Vue.js 配合 TailwindCSS。主要负责对话界面（流式输出）、知识库文件管理 UI、上传进度展示。
* **后端 API (Backend)：** Python (FastAPI / Django) 或 Go。Python 在对接 AI 生态（LangChain, LlamaIndex, Agent SDK）上最具优势。
* **数据库 (Database)：**
    * **关系型数据库：** PostgreSQL / MySQL（存储用户信息、文档元数据、聊天历史、工具调用日志）。
    * **向量数据库：** Milvus / Qdrant / Pinecone（存储文本向量，建议 Qdrant 或 Milvus，对混合检索和元数据过滤支持好）。
* **异步任务队列：** Celery + Redis / RabbitMQ。**（专门满足你的需求）** 用户上传文件后，后端立刻返回任务 ID，由 Celery Worker 在后台执行耗时的 PDF 解析、分块、向量化操作，前端通过 WebSocket 或轮询查询任务状态。

#### 2. RAG 与文档处理流水线 (Document Pipeline)
对于法律文档，传统的按字数截断（Chunking）是灾难性的。
* **文档解析 (Parsing)：** 使用 Unstructured.io 或专门的 PDF 解析模型（如 Nougat、Docling）进行版面分析，识别标题、段落、表格。
* **智能分块 (Chunking)：** 采用**按语义和层级分块**。例如，按“第 X 条，第 Y 款”进行切分，并在 Chunk 中保留父级标题的元数据（Metadata），避免上下文丢失。
* **向量化 (Embedding)：** 使用 BGE-Large、OpenAI `text-embedding-3-large` 或专门的法律领域微调 Embedding 模型。

#### 3. Agent 核心架构 (AI Brain)
你可以使用 Claude Agent SDK，或者 LangGraph、LlamaIndex 的 Agent 模块。无论用哪个，底层的运行机制如下：
* **大模型基座 (LLM)：** 推荐使用推理能力极强的模型作为 Agent 的“大脑”（如 Claude 3.5 Sonnet / Opus，或 GPT-4o），它们在工具调用（Tool Calling）和逻辑推理上表现最稳定。
* **路由与规划 (Routing & Planning)：** 用户提问后，Agent 首先分析意图，决定是直接回答、查询私有库，还是需要去外网搜索。
* **检索与重排 (Retrieve & Re-rank)：**
    * **混合检索：** 结合 BM25（关键词匹配，对特定法律术语极其重要）和 Vector（语义匹配）。
    * **重排 (Re-ranking)：** 必须接入 Re-ranker 服务（如 Cohere Re-rank 或 BGE-Reranker）。检索出 Top 20 后，通过重排模型打分，挑选最相关的 Top 5 喂给 LLM，极大降低幻觉。
* **溯源机制 (Citation)：** Agent 在生成回答时，强制要求其标注信息的来源出处（如：*依据《上传文件A》第X页，第Y条...*），这是法律助手的生命线。

### 研发落地建议

1.  **MVP（最小可行性产品）阶段：** 先不要做大而全的 Agent，优先跑通 **“准确的文档解析 + 混合检索 + 重排 + LLM 总结”** 这个基础的 RAG 链路。法律文件的解析质量决定了最终效果的上限。
2.  **数据安全与隐私：** 法律文件极其机密。需要确保用户的文档不会被用于大模型厂商的训练（使用企业级 API 而非网页版），且不同用户的向量库必须严格做租户隔离（Multi-tenancy）。

### 律师 AI Agent 助手：深度需求与 UX 设计

#### 1. 产品形态与空间布局 (Workspace UI/UX)
律师的工作是基于“案件（Case）”或“项目（Project）”的。因此，系统不能是一个简单的全局聊天框，而必须采用**“工作空间/案卷夹”**的概念。

* **案卷级隔离：** 用户可以创建多个案卷（如“A公司并购案”、“张三劳动纠纷案”）。每个案卷有独立的知识库、独立的 Agent 记忆上下文。
* **经典双屏交互 (Dual-Pane UI)：** 这是法律软件的灵魂。
    * **左侧（占 40%）：** Agent 对话区与工具执行状态栏。
    * **右侧（占 60%）：** 沉浸式文档阅读与编辑区。
* **交互细节：** 当左侧 Agent 回答“依据《XX合同》第 5.1 条”时，这段文字必须是**可点击的超链接（Citation 锚点）**。点击后，右侧自动打开该 PDF 文件，并**高亮定位**到具体的段落。律师需要“眼见为实”才能放心使用。

#### 2. 知识库上传与异步处理体验 (Async Processing UX)
法律卷宗动辄几百页，包含大量扫描件。上传后的处理状态必须对用户绝对透明，缓解等待焦虑。

* **多格式与 OCR 自动触发：** 支持 PDF（纯文本及扫描件）、Word、TXT、图片。系统需自动判断是否为扫描件，如果是，则静默调用 OCR 工具。
* **精细化的状态流转：** 上传后，文件列表展示进度条和具体状态，而不是简单的“处理中”。
    * `排队中 (Queued)`
    * `OCR 识别中 (Extracting Text)`
    * `版面解析中 (Parsing Layout)` —— 识别标题、段落、表格。
    * `语义切块与入库中 (Vectorizing)`
    * `就绪 (Ready)` / `解析失败 (Failed)`
* **人工纠错入口：** 如果某份复杂合同解析乱码，允许用户在右侧预览区手动调整文本或重新指定标题层级。
* **元数据打标 (Metadata Tagging)：** 上传时，允许用户勾选或由 AI 自动提取标签（如：原告证据、被告证据、合同文本、裁判文书），这为后续的“精准检索”提供强大的过滤条件。

#### 3. Agent 核心能力与对话体验细节
这里的重点是让用户感受到 Agent 的“思考过程”，建立信任感。

* **透明的思考链 (Visible Thought Process)：**
    * 用户提问后，对话框上方应出现折叠的“思考状态”，例如：
        * *✅ 正在分析提问意图... 决定检索案卷 A。*
        * *🔍 正在调用【本地知识库检索】，提取关键词：违约金、不可抗力。*
        * *🌐 本地信息不足，正在调用【网络搜索】查询最新司法解释...*
        * *✍️ 正在综合信息生成回答...*
* **多轮追问与沙盒推演：** 允许用户基于某一个回答发起分支对话（例如：“如果对方主张诉讼时效已过呢？”），Agent 能结合刚刚的回答和案卷材料进行沙盒推演。
* **预设专业 Prompt 快捷键：** 聊天框输入 `/` 呼出快捷指令。律师不需要每次都想怎么提问。
    * `/摘要`：生成当前案卷的核心争议焦点。
    * `/风险`：审查选中合同的法律风险点，并以表格输出。
    * `/检索`：仅在库中寻找客观事实，不进行主观推理。

#### 4. 文书/报告撰写与导出 (Drafting & Exporting)
长文本生成是高频刚需，但大模型直接吐出一大段文字体验很差。

* **大纲先行机制：** 对于要求生成“法律意见书”或“尽调报告”的指令，Agent **必须先输出一份大纲**（如：一、案情概述，二、争议焦点，三、法律适用...）。
* **人机协同编辑：** 用户确认或修改大纲后，Agent 再分块生成正文。生成的正文直接输出到右侧的富文本编辑器中，用户可以随时介入修改。
* **专业格式导出：** 支持一键导出为标准的 Word 格式（保留标题层级、字体、页边距等法律文书排版习惯），而非简单的 TXT 转换。

#### 5. 异常流与“防幻觉”兜底机制 (Edge Cases)
* **“找不到依据”的坦白：** 当检索结果相似度极低时，Agent 必须明确回复：“在当前资料库中未检索到相关内容”，**绝对禁止**大模型利用预训练知识进行合理的“编造”。
* **回答置信度提示：** 对于部分模棱两可的推论，Agent 的 UI 侧可显示黄色警告图标，提示律师“该结论基于上下文推理，建议人工核实”。
